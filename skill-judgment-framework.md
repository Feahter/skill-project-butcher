# Skill 沉淀判断标准

*来源：Kimi × 卡兹克 金字塔框架融合*
*时间：2026-03-19*

---

## 一、核心框架：三层分流

遇到新任务时，先判断它属于哪一层：

```
脚本？ → Skill？ → Agent？
```

| 层级 | 特征 | 例子 |
|------|------|------|
| **脚本** | 逻辑固定、输入输出确定、无人介入 | 定时备份文件、格式转换 |
| **Skill** | 需要泛化能力、单一封装、有判断节点 | 网页提取、项目分析、代码审查 |
| **Agent** | 目标导向、步骤未知、需动态规划 | 开发完整脚本、竞品调研、复杂决策 |

**卡兹克原则：好的 Agent 使用方式是不断往下沉淀，让 Agent 只处理还没法被固化的部分。**

---

## 二、Skill 沉淀四维触发器

满足任意一项 → 考虑沉淀；满足两项 → 立即沉淀：

| 维度 | 触发条件 | 权重 |
|------|---------|------|
| **频率** | 同一模式出现 ≥3 次 | 25% |
| **认知负载** | 每次执行需要回忆/查资料 >5 分钟 | 25% |
| **错误成本** | 做错的修复成本 > 执行成本 ×3 | 25% |
| **复利空间** | 可在 ≥3 个场景复用 | 25% |

---

## 三、Skill vs Script 边界判定

| 特征 | 沉淀为 Script | 沉淀为 Skill |
|------|:---:|:---:|
| 输入确定性 | 固定参数，无歧义 | 需解析意图，多路径 |
| 决策点 | 0-1 个 | ≥2 个需判断节点 |
| 输出形态 | 单一结果 | 结构化可扩展 |
| 成长性 | 冻结，改即重写 | 可追加规则和案例 |
| 人机协作 | 完全无人介入 | 需确认/反馈/修正 |

**快速判断法**：复制粘贴能直接跑 → Script；需要口头解释怎么用 → Skill

---

## 四、快速决策检查清单

```
□ 这个问题/模式上周遇到过吗？
□ 能不查资料就写出完整步骤吗？
□ 交给实习生出错概率 > 30% 吗？
□ 解决后有 ≥3 个地方能套用同样思路吗？
□ 6 个月后回看，这个沉淀还有价值吗？
```

- **≥3 个 √** → 立即沉淀为 Skill
- **=2 个 √** → 写 Script，标记观察
- **≤1 个 √** → 临时处理，不沉淀

---

## 五、反模式：什么不该沉淀

| 类型 | 特征 | 处理 |
|------|------|------|
| 过早抽象 | 只出现 1 次就预判未来 | 记入 `observations/`，达 3 次再升级 |
| 领域特化 | 绑定特定业务/人名/版本号 | 剥离上下文，或降级为 Script |
| 快速腐烂 | 依赖月级变更的外部 API | 不沉淀，或标记 `@volatile` |
| 判断过载 | 需要大量实时信息才能决策 | 保持人工，Skill 只沉淀「信息收集框架」 |

## 六、与现有体系的接口

```
scripts/
└── 待观察/        ← 新捕获的候选
    └── YYYY-MM-DD_xxx.sh

SKILL.md 元信息区（每个 Skill 的四维评分）:
  触发频率: N次/月
  认知负载: 高/中/低（需查几分文档）
  错误成本: 高/中/低
  复利空间: 高/中/低
```

**触发记录时机**：
- 每次遇到「这个问题又出现了」
- 每次花了 >10 分钟回忆上次是怎么做的
- 每次踩坑后想「下次不能再这样」
